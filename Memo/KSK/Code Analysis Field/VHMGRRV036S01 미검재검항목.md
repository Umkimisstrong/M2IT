# 미검재검항목 처리 — C# 전환 구조 설계

전제: DevExpress WinForms, 검진항목 리스트는 **DataTable**, TreeList는 **DataSource 바인딩** 방식.

---

## 1. 선행 작업 (변환 전 반드시 확정/준비해야 할 것)

### 1-1. 트리 DataTable에 계층 키 컬럼 추가

Delphi의 `cxTreeList`는 `Level`(0/1) 기반의 가상 계층 구조라 별도 키 컬럼이 없었지만, DevExpress WinForms `TreeList`를 **DataSource 바인딩**으로 쓰려면 자기참조(Self-Referencing) 구조가 필요합니다.

기존 검진항목 DataTable에 아래 2개 컬럼이 없다면 추가해야 합니다.

- `NodeKey` (고유키, 예: Guid 또는 자동증가 int)
- `ParentKey` (부모 로우의 NodeKey, 루트는 null/0)

→ `treeList.KeyFieldName = "NodeKey"`, `treeList.ParentFieldName = "ParentKey"` 로 지정.

### 1-2. 검진항목 DataTable 스키마 확정

`VT_EXAMCD_LIST_A` / 트리 DataTable에 대응되는 컬럼 목록 (Delphi 필드 → 용도):

| Delphi 필드                              | 용도                                  |
| -------------------------------------- | ----------------------------------- |
| VTC_ITEM_GUBUN / 항목구분                  | 항목 분류 (0/4/6 등)                     |
| VTC_ITEM_NO / 검진항목번호                   | 검사코드 (중복 판별 키)                      |
| VTC_EXAM_NAME / 검진항목명                  | 검사명                                 |
| VTC_SEL_GROUP_NO / 선택그룹번호              | 선택그룹                                |
| VTC_GENERAL_SUGA / 일반수가                |                                     |
| VTC_SPC_SUGA / 특검수가                    |                                     |
| VTC_INSURE_CHARGE_SUGA / 의보청구수가        | (원본은 임시로 1 고정 입력 — 실제 값 사용할지 결정 필요) |
| VTC_PRE_ITEM_GUBUN / 이전항목구분            |                                     |
| VTC_PRE_ITEM_NO / 이전검진항목번호             |                                     |
| VTC_PRE_SEL_GROUP_NO / 이전선택그룹번호        |                                     |
| (신규) 단체청구/이전단체청구/개인청구/이전개인청구           | 원본에서 전부 0 고정                        |
| (신규) 수량, DC율, 계약내용, 성별구분, 나이FROM, 나이TO | 원본에서 고정값(1, 0, '') 입력               |

> **확인 필요**: `의보청구수가`, `의보청구금액` 을 원본처럼 `1` 고정값으로 유지할지, 실제 필드값을 쓸지 결정 필요 (원본 주석에도 애매하게 남아있음).

### 1-3. `VT_EXAMCD_LIST_C`의 소유 위치 확인

원본에서 `VT_EXAMCD_LIST_C`는 팝업 폼(`폼_미검재검항목`)이 아니라 **메인 폼 소유**로 보입니다(팝업 코드에 등장하지 않음). C# 전환 시에도 이 리스트는 자식 폼에 넘기지 않고 **메인 폼에서만 직접 사용**하는 것으로 확정할지 확인 필요.

### 1-4. 루트 노드 존재 보장 여부 확정

원본은 `RA_FindRootNode(..., '4')` 결과가 `null`일 가능성을 체크하지 않습니다(무조건 존재 가정). C#에서도 동일하게 "항목구분='4' 루트 로우는 항상 존재한다"고 가정할지, 방어 코드(없으면 생성)를 넣을지 결정 필요.

### 1-5. 이벤트 억제/복원 대상 확정

원본은 `라디오그룹_내시경구분.Properties.OnPropertiesChanged`를 끄고 `OnEditValueChanged`를 다시 켜는 **서로 다른 이벤트**를 사용합니다. C# 쪽에서 실제로 억제해야 하는 이벤트가 무엇인지(둘 다? 하나만?) 원본 폼 설계 의도를 확인해야 합니다.

## 2. 파라미터 정의 (메인 폼 → 자식 폼)

자식 폼(`폼_미검재검항목` → 예: `FrmUnexamRetestItems`) 생성 시 전달해야 할 값:

|파라미터명|타입|설명|
|---|---|---|
|`AcptNo`|`string`|접수번호 (원본 `에디트_접수번호.Text`)|
|`ExamcdListA`|`DataTable`|현재 확정된 검진항목 리스트의 **복사본**(`.Copy()`) — 원본 훼손 방지|

> `VT_EXAMCD_LIST_C`는 (1-3 확정 시) 자식 폼에 넘기지 않음.

## 3. 반환값 정의 (자식 폼 → 메인 폼)

|항목|타입|설명|
|---|---|---|
|`DialogResult`|`DialogResult`|`OK`일 때만 아래 처리 진행|
|`ExamcdListA` (수정본)|`DataTable`|자식 폼에서 체크/이동 작업이 반영된 최종본. 메인 폼의 `VT_EXAMCD_LIST_A`를 이 값으로 교체|

## 4. 메인 폼 후처리 로직에 필요한 것

### 4-1. 삭제 대상 정의

원본: `Level = 1` AND `항목구분 = '4'` 인 노드 전체 삭제. DataSource 바인딩 기준으로는 → **`ParentKey` = (항목구분='4' 루트 로우의 NodeKey)** 인 자식 로우를 전부 삭제.

### 4-2. 추가 로직 (2단계, 순서대로)

1. `ExamcdListA` 순회 → `VTC_ITEM_GUBUN = '4'` 인 로우만 대상
    - 대상 루트: 항목구분='4' 로우
    - 중복 판별: 루트의 기존 자식 중 `검진항목번호`(VTC_ITEM_NO) 동일한 것이 있으면 skip, 없으면 추가
    - 신규 로우의 `항목구분`은 원본 값(`VTC_ITEM_GUBUN`) 그대로 사용
2. `ExamcdListC` 순회 → `VTC_ITEM_GUBUN = '6'` 인 로우만 대상 (// 20241217 대체전 추가)
    - 대상 루트: **동일하게 항목구분='4' 루트**
    - 중복 판별 기준 동일 (검진항목번호)
    - 신규 로우의 `항목구분`은 **강제로 '4'** (원본 값 무시)
    - `이전항목구분/이전검진항목번호/이전선택그룹번호`는 전부 빈 값으로 고정 (원본 그대로)

### 4-3. 신규 로우 필드 매핑 (공통, 두 단계 동일 패턴)

```
항목구분          ← (1단계: 원본값 그대로) / (2단계: 강제 '4')
검진항목번호      ← VTC_ITEM_NO
검진항목명        ← VTC_EXAM_NAME
선택그룹번호      ← VTC_SEL_GROUP_NO
단체청구/이전단체청구/개인청구/이전개인청구 ← 0 고정
일반수가          ← VTC_GENERAL_SUGA
특검수가          ← VTC_SPC_SUGA
의보청구수가      ← 1 고정 (§1-2 확인 필요)
의보청구금액      ← 1 고정 (§1-2 확인 필요)
수량              ← 1 고정
DC율              ← 0 고정
계약내용/성별구분/나이FROM/나이TO ← '' 고정
이전항목구분/이전검진항목번호/이전선택그룹번호
  ← (1단계: VTC_PRE_* 그대로) / (2단계: 전부 '' 고정)
```

### 4-4. 커서 전진 누락 버그 처리 방침 (미정 상태)

원본은 중복이면 레코드 커서가 전진하지 않는 버그가 있습니다. **결정이 없으므로 기본 설계는 "중복 여부와 무관하게 다음 로우로 진행"으로 잡아두고, 실제 구현 단계에서 재확인**하는 것으로 남겨둡니다.

### 4-5. 부가 처리

- 트리/그리드에 합계·건수 등을 별도로 표시하는 라벨이 있다면 반영 후 갱신 필요 여부 확인
- 이벤트 억제 구간: §1-5 결정 후 `try/finally`로 감싸 원상 복구 보장

---

## 5. 처리 순서 (요약)

1. 자식 폼 생성, `AcptNo` / `ExamcdListA.Copy()` 전달
2. `ShowDialog()`
3. 결과가 OK 아니면 종료
4. (OK) 이벤트 핸들러 임시 해제
5. 메인 `ExamcdListA`를 자식 폼 반환값으로 교체
6. 트리 DataTable에서 항목구분='4' 루트의 자식 로우 전체 삭제
7. `ExamcdListA` 중 GUBUN='4' → 루트 하위 추가 (중복 skip)
8. `ExamcdListC` 중 GUBUN='6' → 동일 루트 하위 추가, 항목구분 '4' 강제 (중복 skip)
9. 이벤트 핸들러 복원
10. (필요 시) 합계/건수 등 후처리 갱신
